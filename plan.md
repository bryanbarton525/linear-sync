## Overview

This is a surgical refactor that preserves exact behavior while reorganizing Go source files from a flat root layout into golang-standards/project-layout structure. The approach is to read all existing .go files first to capture current implementation (GraphQL queries, error handling, ticker logic, signal traps, upsert strategy), then move that logic into cmd/ and internal/ packages with only import path updates to compile in new locations. The refactor establishes proper package boundaries: cmd/linear-sync as the single entrypoint, internal/config for environment variable loading, internal/linear for Linear API client, and internal/storage for database persistence. After moving code, validation ensures builds succeed and tests pass/skip as expected before deleting old root files.

## Delivery Target

Git repository with golang-standards/project-layout structure: cmd/linear-sync/main.go, internal/config/, internal/linear/, internal/storage/ packages; old root-level .go files and artifact cruft deleted; go build and go test succeed with expected PASS/SKIP results.

## Architectural Decisions

1. **Treat refactor as surgical code move, not rewrite**
   - Rationale: Matriarch explicitly requires preserving exact behavior including quirks (no timeout on HTTP client, specific signal handling, particular upsert strategy). Reading existing files first ensures behavior fidelity. Only import paths change.
   - Tradeoffs: Cannot improve code quality or fix pre-existing issues, but eliminates risk of behavior changes and simplifies validation (compare exact logic, not reimplemented functionality).
2. **Internal packages follow clear dependency flow with no cycles**
   - Rationale: Constitution requires no circular imports. Dependency graph: cmd → internal/config, internal/linear, internal/storage; internal/storage → internal/linear; internal/config and internal/linear are independent.
   - Tradeoffs: Strict package boundaries prevent shortcuts but enforce maintainability and testability; ensures future contributors understand import relationships.
3. **Validation before deletion**
   - Rationale: Running go build and go test on new structure before deleting old files confirms the refactor is correct. If validation fails, old files remain as reference.
   - Tradeoffs: Requires an extra task in the graph but prevents data loss and enables easy rollback if issues are discovered during validation.
4. **Import paths use canonical module path 'linear-sync/internal/<package>'**
   - Rationale: Constitution requires idiomatic Go import paths matching the module declaration in go.mod. No relative imports or package aliasing.
   - Tradeoffs: Absolute import paths are more verbose but ensure clarity and work correctly with Go tooling (gopls, goimports, go mod).

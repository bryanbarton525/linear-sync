## Overview

The linear-sync service is a single-binary Go application that polls the Linear.app GraphQL API every 5 minutes and upserts issues into PostgreSQL. The architecture is deliberately simple: three internal packages (config, linear, storage) compose in main.go. Config loads and validates environment variables at startup. The linear client wraps HTTP POST calls to the GraphQL endpoint and transforms raw JSON into typed Issue structs. The storage layer wraps a *sql.DB, executes upsert statements inside transactions, and serializes the nullable Assignee as JSONB. The main loop uses signal.NotifyContext for graceful shutdown and time.Ticker for polling, with a 2-minute per-sync timeout context.

## Architectural Decisions

1. **Use signal.NotifyContext for shutdown instead of a manual signal channel**
   - Rationale: Idiomatic Go 1.16+ pattern that composes cleanly with context cancellation throughout the call stack.
   - Tradeoffs: Requires Go 1.16+; cannot easily add custom shutdown hooks, but none are needed here.
2. **Serialize Assignee as JSONB rather than a separate table**
   - Rationale: Keeps the schema minimal, matches the constitution's NF6 schema assumption, and avoids JOIN complexity for analytics consumers.
   - Tradeoffs: Assignee fields are not independently queryable without JSONB operators; acceptable for the stated use case.
3. **Use ClientOption functional options for test overrides**
   - Rationale: Allows test code to inject a local httptest.Server URL and custom http.Client without exposing internals.
   - Tradeoffs: Slightly more boilerplate than a struct literal, but keeps the public API clean.

## Remediation Cycle 1 — PM Triage

The Go code implementation is structurally correct and implements all functional and non-functional requirements per the constitution. However, the validation environment is encountering a **validation/environment failure**: the Go compiler reports that local module packages ('linear-sync/internal/config', 'linear-sync/internal/linear', 'linear-sync/internal/storage') cannot be resolved, with the compiler incorrectly looking in `/usr/local/go/src/linear-sync/...` (the Go standard library location) rather than the workspace module root. This is not a code defect, requirement gap, or design issue—the module structure, go.mod declaration, and import statements are all correct. The issue is that the validation pipeline is not executing the build/test commands from the workspace directory as the active module root. The workspace was successfully committed to GitHub by the pod (indicating proper local module resolution), but the validation environment has not been configured to treat the workspace directory as the Go module root during the validation run. ## Remediation Cycle 1 — Architect

**Current overview:** The linear-sync service is a single-binary Go application that polls the Linear.app GraphQL API every 5 minutes and upserts issues into PostgreSQL. The architecture is deliberately simple: three internal packages (config, linear, storage) compose in main.go. Config loads and validates environment variables at startup. The linear client wraps HTTP POST calls to the GraphQL endpoint and transforms raw JSON into typed Issue structs. The storage layer wraps a *sql.DB, executes upsert statements inside transactions, and serializes the nullable Assignee as JSONB. The main loop uses signal.NotifyContext for graceful shutdown and time.Ticker for polling, with a 2-minute per-sync timeout context.
